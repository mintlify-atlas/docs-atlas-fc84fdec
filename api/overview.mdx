## Introduction

The DitzlerTotes API provides a comprehensive REST interface for managing tote containers, clients, products, locations, and warehouse operations. The API follows modern REST conventions with JSON request/response formats.

## Base URL

All API endpoints are relative to your server's base URL:

```
http://localhost:3000/api
```

In production, replace `localhost:3000` with your deployed server address.

## API Architecture

The DitzlerTotes API follows a modular architecture:

- **Routes** → **Controllers** → **Services** → **Database**
- All database operations use SQL Server stored procedures
- Comprehensive audit logging for all operations
- Support for both JWT and legacy authentication methods

## Request Format

All API requests must include appropriate headers:

```bash
Content-Type: application/json
Authorization: Bearer <your-jwt-token>
```

### Example Request

<CodeGroup>
```bash cURL
curl -X GET "http://localhost:3000/api/health" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

```javascript JavaScript
const response = await fetch('http://localhost:3000/api/health', {
  method: 'GET',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  }
});

const data = await response.json();
```

```python Python
import requests

headers = {
    'Content-Type': 'application/json',
    'Authorization': f'Bearer {token}'
}

response = requests.get('http://localhost:3000/api/health', headers=headers)
data = response.json()
```
</CodeGroup>

## Response Format

All API responses follow a consistent JSON structure:

### Success Response

```json
{
  "success": true,
  "data": {
    // Response data
  },
  "message": "Operation completed successfully"
}
```

### Error Response

```json
{
  "success": false,
  "message": "Error description",
  "error": "Detailed error message"
}
```

## HTTP Status Codes

The API uses standard HTTP status codes:

<ResponseField name="200" type="OK">
  Request completed successfully
</ResponseField>

<ResponseField name="201" type="Created">
  Resource created successfully
</ResponseField>

<ResponseField name="400" type="Bad Request">
  Invalid request parameters or malformed data
</ResponseField>

<ResponseField name="401" type="Unauthorized">
  Missing or invalid authentication token
</ResponseField>

<ResponseField name="403" type="Forbidden">
  Insufficient permissions for the requested operation
</ResponseField>

<ResponseField name="404" type="Not Found">
  Requested resource does not exist
</ResponseField>

<ResponseField name="500" type="Internal Server Error">
  Server error during request processing
</ResponseField>

## Rate Limiting

Currently, the API does not enforce rate limiting. However, we recommend:

- Maximum 100 requests per minute per user
- Implement exponential backoff for failed requests
- Use batch endpoints when available

## Versioning

The API is currently at version 1.0. All endpoints are prefixed with `/api/`.

Future versions will be indicated by route prefixes (e.g., `/api/v2/`).

## Audit Logging

All API requests are automatically logged with:

- User identification (ID, email, role)
- IP address and user agent
- Request timestamp and response time
- Operation type (CREATE, UPDATE, DELETE, VIEW)
- Success/failure status

Audit logs are stored in the database and accessible to administrators.

## Getting Started

<Steps>
  <Step title="Obtain API Credentials">
    Contact your system administrator to receive login credentials.
  </Step>
  
  <Step title="Authenticate">
    Use the `/api/login` endpoint to obtain a JWT token.
    
    See [Authentication](/api/authentication) for details.
  </Step>
  
  <Step title="Make Your First Request">
    Use the token in the `Authorization` header for all subsequent requests.
  </Step>
  
  <Step title="Handle Errors">
    Implement proper error handling based on HTTP status codes.
    
    See [Error Handling](/api/error-handling) for details.
  </Step>
</Steps>
